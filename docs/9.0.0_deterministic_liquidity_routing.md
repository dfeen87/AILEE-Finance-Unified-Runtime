# Layer 9 — Deterministic Liquidity Routing Specification

## 1. Introduction

Layer 9 — Deterministic Liquidity Routing provides a functional, allocator-free mechanism to transfer and route excess asset liquidity under varying stress conditions. It optimizes and scales cross-asset positions based on safety-bounded liquidity maps without allocating dynamic memory on the heap.

The routing engine operates under strict performance and architectural mandates:
* **Zero Heap Allocation:** Purely functional pipeline executing on cache-aligned array storage.
* **Deterministic Flow Convergence:** Given identical initial liquidity states, caps, routing tables, shock bounds, and decisions, the outputs of the routing execution are identical across all environments.
* **Dual-Target Fallback Routing:** Supports routing surplus liquidity to a primary target, falling back to a secondary target if the primary is blocked, and retaining the liquidity at the source if both are restricted.

## 2. Core Structs (64-byte Cache-Aligned)

All primary structures in Layer 9 are cache-line aligned and assert size constraints of exactly 64 bytes:

```cpp
namespace AILLE {

struct alignas(64) LiquidityCap {
    AssetId asset_id;                 // 2 bytes
    uint8_t stress_level;             // 1 byte (0: NORMAL, 1: ELEVATED, 2: CRISIS)
    uint8_t _padding1[5];             // 5 bytes alignment
    double max_outflow_ratio;         // 8 bytes (fraction of current asset value allowed to flow out)
    double max_inflow_ratio;          // 8 bytes (fraction of current asset value allowed to flow in)
    uint8_t reserved[40];             // 40 bytes padding -> 64 bytes total
};
static_assert(sizeof(LiquidityCap) == 64, "LiquidityCap must be exactly 64 bytes");

struct alignas(64) RoutingRule {
    AssetId source;                   // 2 bytes
    AssetId primary_target;           // 2 bytes
    AssetId fallback_target;          // 2 bytes
    uint8_t stress_level;             // 1 byte (0: NORMAL, 1: ELEVATED, 2: CRISIS)
    uint8_t _padding1[1];             // 1 byte alignment
    double preferred_flow_ratio;      // 8 bytes (multiplier for the transfer flow)
    uint8_t reserved[48];             // 48 bytes padding -> 64 bytes total
};
static_assert(sizeof(RoutingRule) == 64, "RoutingRule must be exactly 64 bytes");

struct alignas(64) ShockBounds {
    double max_portfolio_liquidity_shift_per_step; // 8 bytes (max fraction of total portfolio value allowed to shift)
    double max_asset_liquidity_shift_per_step;     // 8 bytes (max fraction of individual asset liquidity allowed to shift)
    uint8_t reserved[48];                          // 48 bytes padding -> 64 bytes total
};
static_assert(sizeof(ShockBounds) == 64, "ShockBounds must be exactly 64 bytes");

struct alignas(64) CrossAssetDecision {
    AssetId asset_id;                 // 2 bytes
    uint8_t flags;                    // 1 byte
    uint8_t _padding1[5];             // 5 bytes alignment
    double target_allocation_ratio;   // 8 bytes
    double current_allocation_ratio;  // 8 bytes
    uint8_t reserved[40];             // 40 bytes padding -> 64 bytes total
};
static_assert(sizeof(CrossAssetDecision) == 64, "CrossAssetDecision must be exactly 64 bytes");

struct alignas(64) StressProfile {
    double volatility_index;          // 8 bytes
    double drawdown_index;            // 8 bytes
    double correlation_index;         // 8 bytes
    uint8_t stress_level;             // 1 byte
    uint8_t reserved[39];             // 39 bytes padding -> 64 bytes total
};
static_assert(sizeof(StressProfile) == 64, "StressProfile must be exactly 64 bytes");

struct alignas(64) LiquidityState {
    AssetId asset_id;                 // 2 bytes
    uint8_t flags;                    // 1 byte (bit 0: frozen/blocked target)
    uint8_t _padding1[5];             // 5 bytes alignment
    double current_liquidity_value;   // 8 bytes
    double current_allocation_ratio;  // 8 bytes
    uint8_t reserved[40];             // 40 bytes padding -> 64 bytes total
};
static_assert(sizeof(LiquidityState) == 64, "LiquidityState must be exactly 64 bytes");

struct alignas(64) LiquidityFlow {
    AssetId source;                   // 2 bytes
    AssetId target;                   // 2 bytes
    uint8_t _padding1[4];             // 4 bytes alignment
    double amount;                    // 8 bytes
    uint8_t reserved[48];             // 48 bytes padding -> 64 bytes total
};
static_assert(sizeof(LiquidityFlow) == 64, "LiquidityFlow must be exactly 64 bytes");

struct alignas(64) RoutingTraceStep {
    AssetId source;                   // 2 bytes
    AssetId target;                   // 2 bytes
    uint8_t stress_level;             // 1 byte
    uint8_t reserved_flags;           // 1 byte
    uint8_t _padding1[2];             // 2 bytes alignment
    double proposed_flow;             // 8 bytes
    double actual_flow;               // 8 bytes
    char log[40];                     // 40 bytes compact descriptive log string -> 64 bytes total
};
static_assert(sizeof(RoutingTraceStep) == 64, "RoutingTraceStep must be exactly 64 bytes");

struct alignas(64) RoutingResult {
    double total_shift_value;         // 8 bytes
    uint32_t flow_count;              // 4 bytes
    uint8_t reserved[52];             // 52 bytes padding -> 64 bytes total
};
static_assert(sizeof(RoutingResult) == 64, "RoutingResult must be exactly 64 bytes");

} // namespace AILLE
```

## 3. Core Algorithm (`route_liquidity`)

The entry point function `route_liquidity` processes initial allocations and flows through a deterministic 4-stage pipeline:

```cpp
DetailedRoutingResult route_liquidity(
    const CrossAssetDecisions& decisions,
    const StressProfile& stress,
    const LiquidityStateSet& states,
    const LiquidityCaps& caps,
    const RoutingTable& table,
    const ShockBounds& bounds
);
```

### Stage 1: Movable Liquidity Calculation
For each asset in the portfolio, the surplus value relative to its target allocation ratio is calculated. The total movable value is clamped to the maximum allowed outflow ratio defined in the asset's active `LiquidityCap` for the current stress level:
$$ \text{surplus\_value} = \max(0.0, \text{current\_ratio} - \text{target\_ratio}) \times \text{total\_portfolio\_value} $$
$$ \text{movable} = \min(\text{surplus\_value}, \text{current\_liquidity} \times \text{max\_outflow\_ratio}) $$

### Stage 2: Dual-Target Rule Evaluation
For each source asset with positive movable liquidity, the active rule in the `RoutingTable` is evaluated:
* **Primary Target:** Attempt to route `movable * preferred_flow_ratio` to the primary target.
* **Fallback Target:** If the primary target is blocked (determined by the target asset's frozen flag or by exceeding the target's maximum inflow cap), attempt to route to the fallback target.
* **Blockage Resolution:** If both targets are blocked, no flow is routed, and the liquidity remains with the source asset.

### Stage 3: Shock Bounds Clamping
To prevent liquidity shock waves, proposed inflows and outflows are capped globally and individually:
* **Asset-Level Shock Bounds:** Caps net changes to any asset to the maximum individual asset shift per step:
  $$ \text{max\_asset\_shift} = \text{current\_liquidity} \times \text{max\_asset\_liquidity\_shift} $$
  If the net proposed flow exceeds this limit, the contributing flows are proportionally scaled down.
* **Portfolio-Level Shock Bounds:** Caps the sum of all shifts to the maximum portfolio shift per step:
  $$ \text{max\_portfolio\_shift} = \text{total\_portfolio\_value} \times \text{max\_portfolio\_liquidity\_shift} $$
  If the total sum of flows exceeds this limit, all flows are scaled down uniformly.

### Stage 4: Trace Logging
Logs all decisions, proposed flows, final actual flows, and blockage reasons deterministically inside `RoutingTraceStep` structures for absolute audibility and visibility.

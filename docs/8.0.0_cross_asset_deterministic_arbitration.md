# Layer 8 — Cross‑Asset Deterministic Arbitration Specification

## 1. Introduction

Layer 8 — Cross‑Asset Deterministic Arbitration is a formal component of the AILEE Finance Runtime introduced in the 8.7 release. Its purpose is to reconcile existing asset‑level advisory outputs into a single, deterministic cross‑asset decision vector for portfolio allocation without generating new underlying market opinions.

Arbitration is designed under strict performance-critical constraints:
* **Zero Heap Allocation:** No dynamic memory allocation in the arbitration path.
* **Pure Determinism:** Given identical input advisories, identical ladder priorities, and identical scaling rules, the engine guarantees identical allocation decisions and trace logs across all platforms, runtimes, and languages.
* **No Side-Effects:** The layer is purely functional (inputs $\to$ outputs + trace) with zero hidden global or persistent runtime state.
* **ABI-Stable Struct Alignment:** All core structural definitions use exactly 64-byte alignments to align with cache-line sizes, ensuring zero padding mismatches and optimal hardware compilation compatibility.

---

## 2. Input Advisory Schema

The input to Layer 8 consists of a bounded vector of per‑asset advisories produced by lower-layer modules (e.g., BRGAM, ERGAM, CRGAM‑X).

### C++ Representation
```cpp
namespace AILLE {

enum class AssetId : uint16_t {
    CASH = 0,
    BTC = 1,
    ETH = 2,
    GOLD = 3,
    OIL = 4,
    EQUITY_HIGH_RISK = 5
};

enum class AdvisoryFlags : uint16_t {
    NONE = 0,
    HARD_BLOCK = 1,
    SOFT_BLOCK = 2,
    PREFERRED = 4
};

struct alignas(64) Advisory {
    AssetId asset_id;
    uint16_t padding_flags;
    double risk_score;        // Raw risk score [0, 100]
    double safety_level;      // Raw safety level [0.0, 1.0]
    double liquidity_level;   // Raw liquidity level [0.0, 1.0]
    double regulatory_level;  // Raw regulatory level [0.0, 1.0]
    double return_score;      // Projected return score [0.0, 1.0]
    double confidence;        // Confidence in signal [0.0, 1.0]
    uint32_t raw_flags;       // AdvisoryFlags bitmask
    uint8_t _padding[12];     // Pad to exactly 64 bytes
};
static_assert(sizeof(Advisory) == 64, "Advisory must be exactly 64 bytes");

} // namespace AILLE
```

### Schema Field Specification
1. `asset_id`: Categorical asset identifier.
2. `risk_score`: High-volatility metrics. High scores indicate elevated downside volatility.
3. `safety_level`: System safety rating. Low levels trigger safety intervention.
4. `liquidity_level`: Market execution capability rating. Used to dampen position sizes to avoid slippage.
5. `regulatory_level`: Regulatory environment status. Low scores map to compliance limits.
6. `return_score`: Target return/conviction estimate.
7. `confidence`: Model confidence overlay.
8. `raw_flags`: Binary flag flags mapping to explicit overrides (e.g., `HARD_BLOCK`).

## 3. Arbitration Ladder Definition (`LADDER_V1`)

The arbitration process resolves trade‑offs and conflicts by walking a global, static priority ladder. The standard prioritized ladder is versioned as **`LADDER_V1`** and cannot be modified at runtime.

The dimensions are evaluated in the following sequence:
$$\text{Safety} \longrightarrow \text{Liquidity} \longrightarrow \text{Regulatory Compliance} \longrightarrow \text{Risk} \longrightarrow \text{Return}$$

### C++ Structural Representation
```cpp
namespace AILLE {

enum class LadderDimension : uint16_t {
    SAFETY = 0,
    LIQUIDITY = 1,
    REGULATORY = 2,
    RISK = 3,
    RETURN = 4
};

struct Ladder {
    const char* version = "LADDER_V1";
    LadderDimension dimensions[5] = {
        LadderDimension::SAFETY,
        LadderDimension::LIQUIDITY,
        LadderDimension::REGULATORY,
        LadderDimension::RISK,
        LadderDimension::RETURN
    };
};

} // namespace AILLE
```

### Ladder Walk Execution Semantic
1. **Safety Filter:** Discards assets failing absolute safety thresholds ($< 0.35$). Caps marginal safety assets ($0.35 \le \text{safety} < 0.6$) at 50% capacity.
2. **Liquidity Scaling:** Multiplies current weight by the normalized liquidity factor.
3. **Regulatory Compliance:** Zeroes out assets with `HARD_BLOCK` flag. Applies soft cap multipliers ($0.2\times$) for low-compliance assets ($< 0.3$). Multiplies by $1.2\times$ for `PREFERRED` assets.
4. **Risk Dampening:** Dampens weight based on normalized risk ($1.0 - \text{normalized\_risk}$).
5. **Return Scaling:** Multiplies weight by a scaling factor ($0.5 + 0.5 \times \text{return\_score}$) based on return potential.

## 4. Canonical Scaling Rules (`SCALING_RULESET_V1`)

To reconcile heterogeneous assets (e.g., cash, crypto, commodities) into a single comparison frame, the engine applies global scaling rules, versioned as **`SCALING_RULESET_V1`**.

These scaling rules are purely functional, allocator-free, and map all dimensions into a standardized range of $[0.0, 1.0]$.

### Scaling Definitions
1. **Risk Normalization:**
   $$\text{risk\_norm} = \text{clamp}\left(\frac{\text{risk\_score} - r_{\min}}{r_{\max} - r_{\min}}, 0.0, 1.0\right)$$
   Where $r_{\min} = 0.0$ and $r_{\max} = 100.0$.
2. **Liquidity Mapping:**
   Directly maps raw inputs or buckets to the range $[0.0, 1.0]$.
3. **Regulatory Mapping:**
   $$\text{regulatory\_norm} = \text{clamp}(\text{regulatory\_level}, 0.0, 1.0)$$
   Discrete mappings for compliance states:
   * Hard blocked $\to 0.0$
   * Marginally compliant $\to 0.3$
   * Neutral $\to 0.5$
   * Highly preferred $\to 1.0$

### C++ Implementation
```cpp
namespace AILLE {

struct ScalingRules {
    const char* version = "SCALING_RULESET_V1";

    static constexpr double clamp(double val, double min, double max) noexcept {
        return (val < min) ? min : ((val > max) ? max : val);
    }

    static constexpr double normalize_risk(double raw_risk, double r_min = 0.0, double r_max = 100.0) noexcept {
        return clamp((raw_risk - r_min) / (r_max - r_min), 0.0, 1.0);
    }

    static constexpr double normalize_liquidity(double level) noexcept {
        return clamp(level, 0.0, 1.0);
    }

    static constexpr double normalize_regulatory(double level) noexcept {
        return clamp(level, 0.0, 1.0);
    }
};

} // namespace AILLE
```

## 5. Output Decision Schema

Arbitration produces a deterministic allocation decision vector and an immutable, compact log of execution.

```cpp
namespace AILLE {

struct alignas(64) AllocationDecision {
    AssetId asset_id;
    double recommended_allocation; // Deterministic allocation fraction [0.0, 1.0]
    uint8_t _padding[54];          // Pad to exactly 64 bytes
};
static_assert(sizeof(AllocationDecision) == 64, "AllocationDecision must be exactly 64 bytes");

struct alignas(64) ArbitrationTraceStep {
    AssetId asset_id;
    uint16_t dimension;            // Dimension evaluated
    double input_value;            // Scale input
    double result_weight;          // Resulting weight
    char log[36];                  // Compact descriptive string
};
static_assert(sizeof(ArbitrationTraceStep) == 64, "ArbitrationTraceStep must be exactly 64 bytes");

struct ArbitrationTrace {
    static constexpr size_t MAX_STEPS = 64;
    ArbitrationTraceStep steps[MAX_STEPS];
    size_t step_count = 0;
};

struct ArbitrationResult {
    static constexpr size_t MAX_ASSETS = 16;
    AllocationDecision decisions[MAX_ASSETS];
    size_t decision_count = 0;
    ArbitrationTrace trace;
};

} // namespace AILLE
```

## 6. Pseudo‑code Implementation

Below is the pseudo‑code representing the pure functional Layer 8 arbitration algorithm.

```text
Function arbitrate(advisories: Advisory[], ladder: Ladder, rules: ScalingRules) -> ArbitrationResult:
    result = ArbitrationResult()
    asset_count = min(advisories.length, MAX_ASSETS)
    result.decision_count = asset_count

    weights = FloatArray(size=asset_count, default=1.0)
    safety_hurdles = FloatArray(size=asset_count)
    liquidity_factors = FloatArray(size=asset_count)
    reg_caps = FloatArray(size=asset_count)
    risk_scores = FloatArray(size=asset_count)
    return_scores = FloatArray(size=asset_count)

    // Phase 1: Canonical Scaling Ruleset (SCALING_RULESET_V1)
    For i from 0 to asset_count - 1:
        adv = advisories[i]
        result.decisions[i].asset_id = adv.asset_id

        safety_hurdles[i] = rules.clamp(adv.safety_level, 0.0, 1.0)
        liquidity_factors[i] = rules.normalize_liquidity(adv.liquidity_level)
        reg_caps[i] = rules.normalize_regulatory(adv.regulatory_level)
        risk_scores[i] = rules.normalize_risk(adv.risk_score)
        return_scores[i] = rules.clamp(adv.return_score, 0.0, 1.0)

    // Phase 2: Ladder Traversal (LADDER_V1)
    For Each dimension In ladder.dimensions:
        For i from 0 to asset_count - 1:
            If weights[i] == 0.0 Then Continue

            adv = advisories[i]

            If dimension == SAFETY Then:
                If safety_hurdles[i] < 0.35 Then:
                    weights[i] = 0.0
                    result.trace.add_step(adv.asset_id, SAFETY, safety_hurdles[i], 0.0, "Safety failure")
                Else If safety_hurdles[i] < 0.6 Then:
                    weights[i] = weights[i] * 0.5
                    result.trace.add_step(adv.asset_id, SAFETY, safety_hurdles[i], weights[i], "Marginal safety cap")
                Else:
                    result.trace.add_step(adv.asset_id, SAFETY, safety_hurdles[i], weights[i], "Safety pass")

            Else If dimension == LIQUIDITY Then:
                weights[i] = weights[i] * liquidity_factors[i]
                result.trace.add_step(adv.asset_id, LIQUIDITY, liquidity_factors[i], weights[i], "Liquidity scaled")

            Else If dimension == REGULATORY Then:
                If (adv.raw_flags & HARD_BLOCK) Then:
                    weights[i] = 0.0
                    result.trace.add_step(adv.asset_id, REGULATORY, 0.0, 0.0, "Hard block")
                Else If reg_caps[i] < 0.3 Then:
                    weights[i] = weights[i] * 0.2
                    result.trace.add_step(adv.asset_id, REGULATORY, reg_caps[i], weights[i], "Reg soft cap")
                Else If (adv.raw_flags & PREFERRED) Then:
                    weights[i] = weights[i] * 1.2
                    result.trace.add_step(adv.asset_id, REGULATORY, reg_caps[i], weights[i], "Preferred mult")
                Else:
                    result.trace.add_step(adv.asset_id, REGULATORY, reg_caps[i], weights[i], "Reg neutral")

            Else If dimension == RISK Then:
                risk_damp = 1.0 - risk_scores[i]
                weights[i] = weights[i] * risk_damp
                result.trace.add_step(adv.asset_id, RISK, risk_scores[i], weights[i], "Risk dampened")

            Else If dimension == RETURN Then:
                return_factor = 0.5 + 0.5 * return_scores[i]
                weights[i] = weights[i] * return_factor
                result.trace.add_step(adv.asset_id, RETURN, return_scores[i], weights[i], "Return scaled")

    // Phase 3: Deterministic Normalization
    total_weight = sum(weights)
    For i from 0 to asset_count - 1:
        final_alloc = 0.0
        If total_weight > 0.0 Then:
            final_alloc = weights[i] / total_weight
        result.decisions[i].recommended_allocation = final_alloc

    Return result
```

## 7. End‑to‑End Walkthrough

To demonstrate the deterministic properties of Layer 8 under `SCALING_RULESET_V1` and `LADDER_V1`, we present a dry run using four heterogeneous advisories corresponding to actual AILEE runtime modules.

### Input Advisories
1. **BTC (BRGAM):**
   * Asset ID: `1`
   * Risk Score: `45.0` (Normalized $\to 0.45$)
   * Safety Level: `0.85`
   * Liquidity Level: `0.90`
   * Regulatory Level: `0.50`
   * Return Score: `0.70`
   * Flags: `NONE`
2. **ETH (ERGAM):**
   * Asset ID: `2`
   * Risk Score: `55.0` (Normalized $\to 0.55$)
   * Safety Level: `0.50` (Marginal safety)
   * Liquidity Level: `0.80`
   * Regulatory Level: `0.25` (Low compliance)
   * Return Score: `0.60`
   * Flags: `NONE`
3. **Gold (CRGAM-X):**
   * Asset ID: `3`
   * Risk Score: `15.0` (Normalized $\to 0.15$)
   * Safety Level: `0.95`
   * Liquidity Level: `0.60`
   * Regulatory Level: `0.90`
   * Return Score: `0.30`
   * Flags: `PREFERRED`
4. **Equity High-Risk:**
   * Asset ID: `5`
   * Risk Score: `90.0` (Normalized $\to 0.90$)
   * Safety Level: `0.30` (Safety failure)
   * Liquidity Level: `0.10`
   * Regulatory Level: `0.40`
   * Return Score: `0.95`
   * Flags: `HARD_BLOCK`

### Step-by-Step Traversal

#### Step 1: Safety Filter
* **BTC:** Safety level $0.85 \ge 0.6 \to$ **Pass** (Weight $= 1.0$)
* **ETH:** Safety level $0.50 \in [0.35, 0.6) \to$ **Marginal Cap** (Weight $= 0.5$)
* **Gold:** Safety level $0.95 \ge 0.6 \to$ **Pass** (Weight $= 1.0$)
* **Equity:** Safety level $0.30 < 0.35 \to$ **Failed** (Weight $= 0.0$, Discarded)

#### Step 2: Liquidity Scaling
* **BTC:** Weight $= 1.0 \times 0.90 = 0.90$
* **ETH:** Weight $= 0.5 \times 0.80 = 0.40$
* **Gold:** Weight $= 1.0 \times 0.60 = 0.60$
* **Equity:** Blocked (Weight $= 0.0$)

#### Step 3: Regulatory Compliance
* **BTC:** Neutral ($0.50$) $\to$ Weight $= 0.90 \times 1.0 = 0.90$
* **ETH:** Low compliance ($0.25 < 0.3$) $\to$ Weight $= 0.40 \times 0.2 = 0.08$ (Soft-capped)
* **Gold:** `PREFERRED` flag $\to$ Weight $= 0.60 \times 1.2 = 0.72$
* **Equity:** Hard block flag $\to$ Weight $= 0.0$

#### Step 4: Risk Dampening
* **BTC:** Risk norm $= 0.45 \to$ Weight $= 0.90 \times (1.0 - 0.45) = 0.495$
* **ETH:** Risk norm $= 0.55 \to$ Weight $= 0.08 \times (1.0 - 0.55) = 0.036$
* **Gold:** Risk norm $= 0.15 \to$ Weight $= 0.72 \times (1.0 - 0.15) = 0.612$

#### Step 5: Return Opportunity Scaling
* **BTC:** Return score $= 0.70 \to$ Factor $= 0.5 + 0.5 \times 0.70 = 0.85 \to$ Weight $= 0.495 \times 0.85 = 0.42075$
* **ETH:** Return score $= 0.60 \to$ Factor $= 0.5 + 0.5 \times 0.60 = 0.80 \to$ Weight $= 0.036 \times 0.80 = 0.0288$
* **Gold:** Return score $= 0.30 \to$ Factor $= 0.5 + 0.5 \times 0.30 = 0.65 \to$ Weight $= 0.612 \times 0.65 = 0.3978$

### Final Allocation Vectors
$$\text{Total Weight} = 0.42075 + 0.0288 + 0.3978 = 0.84735$$

1. **BTC:** $\frac{0.42075}{0.84735} \approx \mathbf{0.49655}$ (49.66%)
2. **ETH:** $\frac{0.02880}{0.84735} \approx \mathbf{0.03399}$ (3.40%)
3. **Gold:** $\frac{0.39780}{0.84735} \approx \mathbf{0.46946}$ (46.94%)
4. **Equity:** $\mathbf{0.00000}$ (0.00%)

This allocation exactly and deterministically reflects safety hurdles, compliance limits, and risk/reward characteristics across the multi-asset scope.

## 8. Maintenance & Lifecycle Governance

### Ladder or Ruleset Version Modifications
The priority sequences defined in `LADDER_V1` and the mathematical mappings of `SCALING_RULESET_V1` are fully compiled structures of the binary. No runtime configuration parsing or hot-swapping is permitted.

Any revision of the priority dimensions or scaling models must be:
1. Formally integrated as a new version constant (e.g., `LADDER_V2` or `SCALING_RULESET_V2`).
2. Subjected to full regression validation under identical golden test vectors.
3. Re-deployed through complete binary release and environment updates.

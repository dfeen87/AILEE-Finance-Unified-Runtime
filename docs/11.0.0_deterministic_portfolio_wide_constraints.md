# Layer 11 — Deterministic Portfolio‑Wide Constraint Engine

## Architecture Specification

Layer 11 acts as the ultimate deterministic, allocator-free guardrail for the AILEE Finance Runtime Kernel. It consumes proposed portfolio allocations and flows (post-governor reconciliation) and applies hard constraints across the entire portfolio. This layer enforces max-exposure bounds, sector caps, pairwise correlation clusters, and risk-budget limits in a deterministic 4-stage pipeline.

---

### 1. Deterministic 4-Stage Execution Pipeline

1. **MAX_EXPOSURE_CLAMP (Stage 1)**
   - Enforces per-asset and directional bounds on target allocations.
   - For long-only or short-supported environments, clamps allocations directly:
     $$ w_i \leftarrow \max(-{\text{max\_short\_exposure}}_i, \min(w_i, {\text{max\_long\_exposure}}_i)) $$

2. **SECTOR_CAP_ENFORCE (Stage 2)**
   - Maps each asset to its canonical sector.
   - Sums absolute allocations within each sector and, if exceeded, proportionally scales back only the sector constituents to satisfy the cap:
     $$ w_i \leftarrow w_i \times \frac{\text{max\_sector\_exposure}_s}{\sum_{j \in \text{sector } s} |w_j|} $$

3. **CORRELATION_DAMPEN (Stage 3)**
   - Identifies highly-correlated clusters where pairwise correlation coefficient $\rho_{ij} > 0.70$.
   - Computes the total allocation of each cluster. If it exceeds `CORR_CLUSTER_MAX_ALLOCATION` (0.30), scales down all cluster assets proportionally:
     $$ w_i \leftarrow w_i \times \frac{0.30}{\sum_{j \in \text{cluster}} |w_j|} $$

4. **RISK_BUDGET_ENFORCE (Stage 4)**
   - Computes the portfolio-wide linear weighted risk metric:
     $$ \text{TotalRisk} = \sum_i |w_i| \cdot \text{RiskScore}_i $$
   - If $\text{TotalRisk} > \text{RiskBudget.max\_risk}$, scales down all non-cash assets proportionally to bring total risk exactly to the budget threshold:
     $$ w_i \leftarrow w_i \times \frac{\text{RiskBudget.max\_risk}}{\text{TotalRisk}} \quad (\text{for } i \neq \text{AssetId::CASH}) $$

### 2. Struct Definitions (64-byte Cache-Aligned)

All data structures are designed to be exactly 64 bytes with explicit padding, satisfying allocator-free performance:

```cpp
struct alignas(64) ConstraintRule {
    AssetId asset_id;                 // 2 bytes
    uint8_t is_active;                // 1 byte
    uint8_t _padding1[5];             // 5 bytes alignment
    double max_long_exposure;         // 8 bytes
    double max_short_exposure;        // 8 bytes
    uint8_t reserved[40];             // 40 bytes padding
};

struct alignas(64) SectorDefinition {
    uint8_t sector_id;                // 1 byte
    uint8_t is_active;                // 1 byte
    uint8_t _padding1[6];             // 6 bytes alignment
    double max_sector_exposure;       // 8 bytes
    char sector_name[16];             // 16 bytes
    uint8_t reserved[32];             // 32 bytes padding
};

struct alignas(64) CorrelationProfile {
    AssetId asset_a;                  // 2 bytes
    AssetId asset_b;                  // 2 bytes
    uint8_t is_active;                // 1 byte
    uint8_t _padding1[3];             // 3 bytes alignment
    double correlation_score;         // 8 bytes
    uint8_t reserved[48];             // 48 bytes padding
};

struct alignas(64) RiskBudget {
    double max_portfolio_risk;        // 8 bytes
    uint8_t is_active;                // 1 byte
    uint8_t _padding1[7];             // 7 bytes alignment
    uint8_t reserved[48];             // 48 bytes padding
};

struct alignas(64) ConstraintTraceStep {
    AssetId asset_id;                 // 2 bytes
    uint8_t stage;                    // 1 byte
    uint8_t action_taken;             // 1 byte
    uint8_t _padding1[4];             // 4 bytes alignment
    double before_value;              // 8 bytes
    double after_value;               // 8 bytes
    char log[40];                     // 40 bytes compact log
};
```

### 3. Walkthrough Example

Consider a portfolio trying to over-allocate to a highly-correlated Crypto Cluster consisting of BTC and ETH:

- **Initial State**:
  - BTC Proposed: `0.35` (Risk Score: `60.0`)
  - ETH Proposed: `0.25` (Risk Score: `80.0`)
  - CASH Proposed: `0.40` (Risk Score: `0.0`)
  - Initial Portfolio Risk = `(0.35 * 60) + (0.25 * 80) = 41.0`

- **Stage 1 (Max Exposure)**:
  - BTC Max Long: `0.40`, ETH Max Long: `0.30`. Neither breaches; allocations remain `0.35` and `0.25`.

- **Stage 2 (Sector Caps)**:
  - Sector `CRYPTO` Cap: `0.50`.
  - Joint Exposure = `0.35 + 0.25 = 0.60` (Breached!).
  - Scale Factor = `0.50 / 0.60 = 5/6`.
  - BTC allocation $\rightarrow$ `0.29167`.
  - ETH allocation $\rightarrow$ `0.20833`.

- **Stage 3 (Correlation Dampening)**:
  - Pairwise Correlation $\rho_{\text{BTC, ETH}} = 0.85$ (exceeds `0.70` threshold).
  - Joint Cluster Exposure = `0.50`.
  - Cluster Max Limit = `0.30`.
  - Scale Factor = `0.30 / 0.50 = 0.60`.
  - BTC allocation $\rightarrow$ `0.175`.
  - ETH allocation $\rightarrow$ `0.125`.

- **Stage 4 (Risk Budget)**:
  - Max Portfolio Risk Budget: `25.0`.
  - Evaluated Portfolio Risk = `(0.175 * 60.0) + (0.125 * 80.0) = 20.5`.
  - `20.5 <= 25.0` (Under budget). No further risk scaling applied.

- **Final State**:
  - BTC allocation: `0.175`
  - ETH allocation: `0.125`
  - CASH allocation: `0.400`
  - Final Risk: `20.5`
  - All constraints satisfied deterministically.
